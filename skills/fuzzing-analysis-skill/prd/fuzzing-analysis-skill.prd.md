# 自动化测试添加 Agent 编排系统 PRD（简版）

## 1. 背景

- 需要建设一套“自动化测试添加系统”。
- 系统要根据用户输入的测试补齐目标，自动分析项目测试缺口，并调度不同 Agent 分阶段完成测试添加任务。
- 系统要支持分析、执行、验证、修复与任务推进的闭环流程。

## 2. 目标

- 构建一个以主 Agent 为调度核心的测试自动添加系统。
- 主 Agent 负责整体流程控制。
- 分析 Agent 负责任务分析与测试缺口识别。
- 执行 Agent 负责具体测试补齐任务。
- 整个任务优先建设通用 Agent 模板，再展开具体 skill 的业务实现。
- 系统要基于 skill 结构来编排，主 Agent、分析 Agent、执行 Agent 分别对应不同的 skill。
- 系统需要具备标准化、可复用、可扩展的 Agent 模板能力，便于后续接入不同类型的 Agent 与任务场景。
- 每个 skill 需要提供自己的说明文件 `SKILL.md`，用于描述能力、参数、入口和输出约定。
- 主 Agent 先读取 skill 的说明文件，再按约定调用对应的入口文件完成任务。
- 所有 skill 共享的接入协议、参数协议、输出协议、状态协议、日志协议和兼容性协议，统一由 `common/` 目录下的公共协议文档定义。

## 3. 角色职责

### 3.1 主 Agent

- 接收用户需求。
- 调度分析 Agent 与执行 Agent。
- 维护任务执行流程。
- 根据执行结果决定是否进入下一阶段任务。

### 3.2 分析 Agent

- 分析项目测试现状。
- 识别需要补充测试的模块或文件。
- 产出可供执行 Agent 使用的阶段性任务。

### 3.3 执行 Agent

- 接收单个阶段任务。
- 完成测试添加、验证、修复与结果反馈。
- 作为通用任务执行单元，不直接承担整体调度职责。

## 4. 核心需求
- 系统需要支持循环式任务推进。
  主 Agent 调用分析 Agent 获取测试添加任务后，将任务拆分为多个阶段，并逐个交给执行 Agent 处理。执行 Agent 完成当前阶段后应反馈任务状态，主 Agent 根据反馈判断是否继续分发下一个任务。
- 系统需要支持标准化 Agent 模板。
  所有 Agent 应遵循统一的框架规范，便于管理任务描述、日志记录、输出结果与状态反馈。
- 系统需要支持测试覆盖率分析。
  分析 Agent 应优先基于覆盖率相关文件进行判断；若相关文件不存在，则使用覆盖率测试结果作为替代依据。

## 5. 通用 Agent 模板需求

- 通用 Agent 模板应作为一层独立的基础能力，提供类似 SDK 的统一接口和运行规范，供不同 Agent 直接复用。
- 通用 Agent 模板不应被写成一个独立 skill，而应作为共享的公共能力放在单独目录中。
- 模板不应绑定某一个具体工具，也不应把基础能力分散到各个 Agent 自身实现中。
- 模板需要覆盖统一的运行入口、任务参数封装、日志记录、状态回传和结果输出等基础能力，确保不同 Agent 在同一套标准下工作。
- 这里的“统一入口”指统一的调用约定，不是所有 skill 共用同一个入口文件。
- 通用 Agent 模板由所有 skill 共享，各个 skill 的脚本直接调用这份公共能力。
- 各个 skill 自身的 `scripts` 只保留本 skill 的私有脚本和适配逻辑，不再复制通用模板。
- 具体业务 skill 不应再内嵌通用能力，而应直接依赖这层公共模板。

## 6. 非目标

- 本阶段不要求实现具体代码。
- 本阶段不限定具体技术方案。
- 本阶段不强制规定 Agent 内部实现细节。
- 本阶段不要求一次性覆盖所有项目类型。
- 当前重点是明确系统边界、角色职责与核心流程。
