# 脚本具体 PRD: 外部SDK升级检查器

## 1. 目标
- 检查当前项目 `src/sdk` 下的外部 SDK 是否落后于 `/home/coder/github/kevinanew/api_sdk/python`
- 只做检查和输出，不修改代码
- 通过主文件对比判断是否需要升级

## 2. 运行约定
- 用法：`python code_doctor/外部SDK升级检查器.py [项目名]`
- 不传参数时，默认使用当前工作目录
- 先更新参考仓库 `api_sdk` 的 `master`：`git fetch origin` -> `git checkout master` -> `git reset --hard origin/master`
- 参考仓库更新失败不阻断后续检查，只记录失败状态

## 3. 检查规则
- 扫描范围：当前项目 `src/sdk` 下所有非隐藏目录
- 排除项：本项目 SDK、`room_sanic`、`__pycache__`、所有以 `.` 开头的文件和目录
- 升级分组用统一策略表管理；每组需要定义：
  - 组内包含哪些 SDK 名称
  - 该组最终统一对齐到哪个参考目录
  - 是否只能在普通 SDK 升级完成后再处理
- 当前必须支持的专属组：
  - `room` / `room_v10` -> `room_sanic`
  - `user_profile` / `user_profile_sanic` -> `user_profile_flask`
- 主文件规则：优先 `目录同名.py`，否则 `__init__.py`
- 判定规则：主文件内容一致则已升级；不一致则未升级
- 无法判断：主文件缺失、读取失败、参考仓库找不到对应 SDK；这类项只列出原因，不作为升级目标
- 处理上限：一次最多处理 3 个未升级 SDK，超过部分只说明数量
- 若当前 `src/sdk` 下已存在正在进行中的 SDK 升级改动，先提示合并，再停止本轮其他 SDK 升级需求
- 必须确保 `sdk_client.py` 和 `test_sdk_client.py` 存在，且只能放在当前项目 `src/sdk/` 根目录

## 4. 架构要求
- 必须使用面向对象组织代码
- 核心能力至少拆成：参考仓库更新、SDK 扫描、状态分析、提示词管理、输出控制
- 禁止把核心流程散落成一堆无组织的顶层 `def`
- 所有提示词模板统一集中管理，位置靠近文件头部，便于维护
- 主流程必须先完成状态分析，再按分析结果选择对应提示词
- 专属升级组只在普通 SDK 升级完成后才进入
- 专属升级组必须能按策略表扩展，不能写死在流程里

## 5. 输出规则
- 无需升级时：输出 `无需要升级的 SDK`，退出码 `0`
- 需要升级时：输出对应的 Markdown 提示词，退出码 `1`
- 出现异常时：输出错误信息，退出码 `1`
- 输出只面向 stdout

## 6. 提示词内容
- 需要处理的 SDK 名称
- 当前项目根目录与参考仓库目录
- 参考仓库 SDK 选择优先级：`_flask` -> 同名 -> `_sanic`
- `sdk_client.py` / `test_sdk_client.py` 只能在 `src/sdk/` 根目录
- 需要对齐主文件、补齐缺失文件、保持目录结构一致、限制修改边界
- `room` 的最终标准实现是 `room_sanic`
- 无法判断的 SDK 必须在输出中单独列出并写明原因
- 每次只生成最多 3 个普通未升级 SDK 的处理提示词
